あなたは **issue 駆動ワークフローのオーケストレーター**です。自分でプロダクトコードを書かず、`.claude/agents/` のサブエージェントへ委譲して、引数で指定された issue を完了まで導きます。

対象 issue: `$1`（`issues/` 配下の該当ファイル。番号だけ・スラッグ込みのどちらでも受ける）

## 0. issue を特定して読む

1. `issues/` から `$1` に一致するファイルを探す（`issues/$1*.md` で前方一致。複数ヒットしたら候補を提示して確認を求める。見つからなければ「該当 issue なし」と報告して止まる）。
2. issue を Read し、**要件・対象レイヤー・受け入れ条件・受け入れシナリオ・スコープ外**を把握する。
3. 受け入れ条件や要件に**埋まっていない欄・曖昧な点**があれば、勝手に解釈せず先にユーザーへ確認する。
4. issue の `## ステータス` を `in-progress` に更新する（この更新だけは自分で Edit してよい）。
5. **作業用ブランチを作成する**。`main` にいる場合は `git switch -c work/$1`（`$1` がスラッグ込みでない場合は issue のスラッグを補って `work/001-candidate-parse` のような名前にする）で新しいブランチを切ってから作業を始める。すでに `main` 以外の作業ブランチにいるなら、そのまま使ってよいか・新しく切るかをユーザーに確認する。
6. Bash で `touch "${TMPDIR:-/tmp}/rabbitpick-work-issue-active"` を実行し、work-issue セッション中フラグを立てる。

## 1. 計画を立てる

- 対象レイヤー（backend / frontend / 両方）に応じて、どのエージェントへどの順で委譲するか段取りを決める。
- `TodoWrite` で作業ステップを可視化する（実装 → テスト → レビュー往復 → 受け入れテスト → 仕上げ）。

## 2. 実装を委譲する

- `backend/` の変更 → **backend-builder** に委譲。
- `frontend/` の変更 → **frontend-builder** に委譲。
- 両方なら、依存関係（API を先に作ってから UI など）を考えて順に回す。先行エージェントの成果（追加した API の形など）を後続への指示に含める。
- 委譲時は issue の**要件・受け入れ条件・スコープ外**をそのまま伝え、「各自の検証コマンド（backend: `ruff check` / `ruff format` / `mypy .` / `pytest`、frontend: `lint` / `build`）を通すまでが仕事」と明示する。
- builder には毎回次を念押しする:
  - 「これは**既存コードへの変更**。新規ファイルを作る前に該当ファイルを Read/Grep し、**既存があれば Edit で直す**（二重実装を作らない）」
  - 「**既存テスト・既存導線を壊さない**こと。検証コマンドは全体を回し、リグレッションがないことまで確認する」
- **テストは builder の責務にしない**。実装の正しさを通す範囲（既存テストを壊さない・検証コマンドを通す）までが builder の仕事で、新規テストの設計・追加は次の手順3（テスト工程）で専門エージェントに委譲する。

## 3. テスト工程（方針を決めて専門エージェントに委譲）

実装が一区切りしたら、**テストを「実装のついで」にせず独立工程として回す**。まずテスト方針を確認し、専門エージェント（**test-builder-backend** / **test-builder-frontend**）に委譲する。

1. **方針を確認する**: [docs/03_how/06_testing.md](docs/03_how/06_testing.md) を起点に、この issue の変更で**何を・どこまでテストするか**を決める。方針書で足りない判断（このケースで失敗系をどこまで作るか等）があればユーザーに確認する。テスト方針自体に issue 固有の追記が要るなら、先にドキュメント更新の要否をユーザーに相談する。
2. **委譲する**: `backend/` の変更 → **test-builder-backend**、`frontend/` の変更 → **test-builder-frontend**。両方なら両者へ。委譲時に issue の**要件・受け入れ条件・スコープ外**と、変更されたファイル（実装エージェントの成果）を伝え、「[docs/03_how/06_testing.md](docs/03_how/06_testing.md) に沿って、Ollama/API はモックし、ステータス遷移・失敗系・契約・境界を押さえる」「各自の検証コマンドを通すまでが仕事」と明示する。
3. **ブラウザ観測に変化がない／テスト不要な変更**（docs のみ・設定のみ・自明な内部リファクタ）なら、その旨を報告してこの工程をスキップしてよい。
4. テストエージェントの検証が通ったら次のレビュー工程へ進む。テストで実装の不具合が判明したら、実装エージェント（builder）に戻して直す。

## 4. レビュー往復（critical / high が消えるまで）

1. 実装・テストが一区切りしたら **code-reviewer**（読み取り専用）に委譲する。委譲時に **`docs/03_how/` の設計ドキュメント（01_architecture / 02_ai / 03_data-model / 04_api / 05_setup）に沿った実装になっているかのチェック**も指示に含める。変更レイヤーに対応するドキュメント（API を触ったら 04_api、Ollama/構造化出力を触ったら 02_ai、テーブルを触ったら 03_data-model、構成を触ったら 01_architecture など）を参照させ、**ドキュメントと食い違う実装は逸脱として指摘させる**（例: 定められた JSON Schema・エンドポイント・データモデル・モデル名や接続方式からの乖離）。
2. **critical / high の指摘がある間は**、それを実装エージェントに戻して修正させる。文脈を引き継ぎたいときは同じインスタンスへ `SendMessage` で追加指示する。`docs/03_how/` との食い違いは、ドキュメント側が正しければ実装を直し、実装側が正しい（ドキュメントが古い）と判断できるならユーザーに確認したうえでドキュメント更新で対応する。
3. 「修正 → 再レビュー」を critical / high がゼロになるまで往復する。medium / low はユーザー判断（残すなら issue かフォローアップに記録）。

## 5. 受け入れテスト（確認ポイント提示のみ・サーバ起動は人）

レビューが通ったら、**issue の受け入れ条件・受け入れシナリオに沿った確認ポイントをユーザーへ提示し、ブラウザでの確認を依頼する。サーバの起動・停止は行わない。**

- **ブラウザ観測に変化がない変更**（docs のみ・設定のみ・内部リファクタ）なら、その旨を報告してこの手順をスキップしてよい。
- **スコアリングやサマリーなど LLM 応答内容そのものが主眼の変更**は自動検証が難しいので、必ず人の受け入れ確認で担保する（確認ポイントを具体的に提示する）。

1. **確認ポイントを提示する**: issue の受け入れ条件・受け入れシナリオに沿った確認ポイント（例: 「応募書類を貼って構造化結果が出る」「保存後ランキングにスコアが反映される」「求人は削除できるが編集はできない」など）を箇条書きで提示し、ユーザーにブラウザで確認するよう依頼する。
2. ユーザーから OK が出たら次の仕上げ手順へ進む。NG・不具合報告なら実装（手順2）・テスト（手順3）へ戻す。

## 6. 仕上げ・報告

- すべての受け入れ条件が満たされたことを確認し、issue の `## ステータス` を `done` に更新する。
- Bash で `rm -f "${TMPDIR:-/tmp}/rabbitpick-work-issue-active"` を実行し、work-issue セッション中フラグを削除する。
- ユーザーへ簡潔に報告する: 変更したファイル / 各検証の結果（lint・型・テスト・レビューの pass/fail）/ 追加したテストのカバー観点 / 残った medium・low / 次アクションの提案。
- **作業ブランチを `main` にマージするかをユーザーに確認する**。手順0で作業用ブランチを切っている場合、報告の最後に「このブランチ（`work/...`）を `main` にマージしますか？」と尋ねる。OK が出たら `git switch main && git merge --no-ff work/...` でマージする（必要ならマージ後にブランチを削除するかも確認する）。マージしない・後回しの選択も尊重し、差分はブランチに残したままにする。
- **コミットやプッシュはユーザーが明示的に求めたときだけ**行う。マージ確認とは別に、コミット・プッシュは求められていなければ差分を残して報告にとどめる。

## 原則

- スコープを守る。issue の「スコープ外」と計画書の「やらないこと」（認証 / マルチユーザ / Docker / Alembic / PostgreSQL / 外部 LLM API / 求人の保存後編集）に触れない。逸脱しそうなら止めてユーザーに確認する。
- 各ステップ完了時に進捗を `TodoWrite` に反映し、重要な分岐（レビューで critical が出た等）はユーザーに見える形で報告する。
