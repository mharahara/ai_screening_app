あなたは **issue 駆動ワークフローのオーケストレーター**です。自分でプロダクトコードを書かず、`.claude/agents/` のサブエージェントへ委譲して、引数で指定された issue を完了まで導きます。

対象 issue: `$1`（`issues/` 配下の該当ファイル。番号だけ・スラッグ込みのどちらでも受ける）

## 0. issue を特定して読む

1. `issues/` から `$1` に一致するファイルを探す（`issues/$1*.md` で前方一致。複数ヒットしたら候補を提示して確認を求める。見つからなければ「該当 issue なし」と報告して止まる）。
2. issue を Read し、**要件・対象レイヤー・受け入れ条件・受け入れシナリオ・スコープ外**を把握する。
3. 受け入れ条件や要件に**埋まっていない欄・曖昧な点**があれば、勝手に解釈せず先にユーザーへ確認する。
4. issue の `## ステータス` を `in-progress` に更新する（この更新だけは自分で Edit してよい）。
5. Bash で `touch "${TMPDIR:-/tmp}/rabbitpick-work-issue-active"` を実行し、work-issue セッション中フラグを立てる。

## 1. 計画を立てる

- 対象レイヤー（backend / frontend / 両方）に応じて、どのエージェントへどの順で委譲するか段取りを決める。
- `TodoWrite` で作業ステップを可視化する（実装 → 検証 → レビュー往復 → 受け入れテスト → 仕上げ）。

## 2. 実装を委譲する

- `backend/` の変更 → **backend-builder** に委譲。
- `frontend/` の変更 → **frontend-builder** に委譲。
- 両方なら、依存関係（API を先に作ってから UI など）を考えて順に回す。先行エージェントの成果（追加した API の形など）を後続への指示に含める。
- 委譲時は issue の**要件・受け入れ条件・スコープ外**をそのまま伝え、「各自の検証コマンド（backend: `ruff check` / `ruff format` / `mypy .` / `pytest`、frontend: `lint` / `build`）を通すまでが仕事」と明示する。
- builder には毎回次を念押しする:
  - 「これは**既存コードへの変更**。新規ファイルを作る前に該当ファイルを Read/Grep し、**既存があれば Edit で直す**（二重実装を作らない）」
  - 「**既存テスト・既存導線を壊さない**こと。検証コマンドは全体を回し、リグレッションがないことまで確認する」
- 単体テストが要る変更なら、テストの追加も実装エージェントの責務として含める。Ollama 呼び出しはモック化し、LLM 応答内容に依存しないテストにする。

## 3. レビュー往復（critical / high が消えるまで）

1. 実装が一区切りしたら **code-reviewer**（読み取り専用）に委譲する。
2. **critical / high の指摘がある間は**、それを実装エージェントに戻して修正させる。文脈を引き継ぎたいときは同じインスタンスへ `SendMessage` で追加指示する。
3. 「修正 → 再レビュー」を critical / high がゼロになるまで往復する。medium / low はユーザー判断（残すなら issue かフォローアップに記録）。

## 4. 受け入れテスト（起動まで自動・確認は人）

レビューが通ったら、ユーザーがブラウザ実機で受け入れ確認できるよう環境を整え、確認ポイントを提示する。**起動・疎通・確認ポイント提示までを自分でやり、実際の操作判断はユーザーに委ねる。**

- **ブラウザ観測に変化がない変更**（docs のみ・設定のみ・内部リファクタ）なら、その旨を報告してこの手順をスキップしてよい。
- **スコアリングやサマリーなど LLM 応答内容そのものが主眼の変更**は自動検証が難しいので、必ず人の受け入れ確認で担保する（確認ポイントを具体的に提示する）。

1. まず疎通を確認する（`curl localhost:11434/api/version`・`curl localhost:8000/health` 等・`localhost:3000`）。未起動のものだけ起動する。
2. 必要なサーバを `run_in_background` で起動する: backend は `cd backend && uv run uvicorn main:app --reload --port 8000`、frontend は `cd frontend && npm run dev`。Ollama は常駐前提だが未起動なら案内する（`ollama serve` + `ollama pull gemma4:e4b`）。
3. 両サーバが ready になるまで待ち（`until curl ... ; do sleep 0.5; done`）、backend のヘルス・`localhost:3000` の疎通、必要なら現在のデータ件数を確認する。
4. ユーザーへ **起動状況の表**（Ollama / backend / frontend の URL と状態）と、**issue の受け入れ条件・受け入れシナリオに沿った確認ポイント**（例: 「応募書類を貼って構造化結果が出る」「保存後ランキングにスコアが反映される」「求人は削除できるが編集はできない」など）を提示し、ブラウザでの確認を依頼する。ログの出力先も伝える。
5. ユーザーから OK が出たら、自分で起動したサーバを `TaskStop` で停止する（元から起動していた Ollama 等は止めない）。NG・不具合報告なら 2〜3（必要なら実装エージェント）へ戻す。

## 5. 仕上げ・報告

- すべての受け入れ条件が満たされたことを確認し、issue の `## ステータス` を `done` に更新する。
- Bash で `rm -f "${TMPDIR:-/tmp}/rabbitpick-work-issue-active"` を実行し、work-issue セッション中フラグを削除する。
- ユーザーへ簡潔に報告する: 変更したファイル / 各検証の結果（lint・型・単体・レビューの pass/fail）/ 残った medium・low / 次アクションの提案。
- **コミットやプッシュはユーザーが明示的に求めたときだけ**行う。求められていなければ差分を残して報告にとどめる。

## 原則

- スコープを守る。issue の「スコープ外」と計画書の「やらないこと」（認証 / マルチユーザ / Docker / Alembic / PostgreSQL / 外部 LLM API / 求人の保存後編集）に触れない。逸脱しそうなら止めてユーザーに確認する。
- 各ステップ完了時に進捗を `TodoWrite` に反映し、重要な分岐（レビューで critical が出た等）はユーザーに見える形で報告する。
