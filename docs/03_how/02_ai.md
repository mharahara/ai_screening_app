# AI 連携の方針 (how)

- **実行基盤**: バックエンドの `services/` 層から LLM provider へ問い合わせる。provider は `LLM_PROVIDER` 環境変数で切り替える（構造化・マッチングとも一括）。
  - `ollama`（デフォルト）: ローカルで起動した Ollama（モデル `gemma4:e4b`）に公式 Python SDK（`ollama`）の `client.chat()` で問い合わせる。`localhost:11434` 常駐前提
  - `claude`: Claude（Anthropic API、モデル `claude-opus-4-8`）に公式 Python SDK（`anthropic`）の `messages.create()` で問い合わせる。`ANTHROPIC_API_KEY` が必要で、API キー・通信コストが発生する
- **provider 抽象**: provider 差分（SDK・構造化出力の指定方法・例外種別）は `services/llm.py` の各 provider 実装に閉じ込め、「検証 + リトライ + フィードバック」の共通ループは provider 非依存に保つ。`services/structuring.py`・将来のマッチング層は provider を意識せず `structured_chat()` を呼ぶだけ
- **構造化出力**: 出力を JSON に固定し、受け取った結果を Pydantic スキーマで検証・パースする。固定方法は provider 差分（ollama は `format`、claude は `output_config.format`）で、いずれも Pydantic 由来の JSON Schema を渡す（claude は Anthropic SDK の `transform_schema` で structured outputs 互換に整形）
- **構造化**: 求人票・応募書類の生テキストを LLM に渡し、定義済みフィールドの JSON を返させる
- **マッチング**: 求人要件と候補者データを LLM に渡し、各軸（スキル / 経験年数 / 業界経験 / ポジションレベル）のスコア、必須要件の yes/no 判定、強み・懸念点・確認事項を JSON で返させる
- **堅牢化**: JSON パース失敗時はリトライする。provider 選択・モデル名・接続先は pydantic-settings 経由で `.env` から読み込み、切り替え可能にする
- **べき等性**: 同じ候補者×求人の再評価は上書き保存する

## 詳細仕様

### 構造化

求人票・応募書類の生テキストを LLM に渡し、定義済みフィールドの JSON を返させる。`POST /jobs/parse` / `POST /candidates/parse` の2エンドポイントで利用し、**未保存の構造化結果のみ**を返す（保存は別エンドポイント）。

#### 基本方針

| 項目 | 方針 |
|------|------|
| 欠損フィールド | null 許容。原文に記載がないフィールドは null（配列は空配列）。原文必須項目以外は Optional |
| 推測 | 推測しない。原文から読み取れない値は埋めず null にする（ハルシネーション防止） |
| 正規化 | LLM で正規化。経験年数→int、単価→数値レンジ、スキル→配列、区分→enum にして返させる |
| 言語 | 入力・出力ともに日本語前提。生成テキスト系フィールドは日本語で返す |
| 原文 | LLM には渡すが構造化対象には含めず、サーバ側で `raw_text` を別途保持する |

#### (a) Pydantic スキーマ定義（`schemas.py`）

保存用スキーマ（`JobCreate` 等）とは別に、構造化結果スキーマ `JobParseResult` / `CandidateParseResult` を定義する。

**enum 定義**

| enum | 候補値 | 補足 |
|------|--------|------|
| `EmploymentType`（雇用形態） | 正社員 / 契約社員 / 業務委託 / 派遣 / その他 | 不明時は null |
| `RemoteWork`（リモート可否） | フルリモート / 一部リモート / 出社 / 不明 | 区分。曖昧だが言及はある場合 `不明`、言及なしは null |
| `PositionLevel`（ポジションレベル） | ジュニア / ミドル / シニア / リード / マネージャー | 原文の役割記述から正規化 |

**JobParseResult（求人）**

| フィールド | 型 | 必須/Optional | 単位・備考 |
|------------|----|--------------|-----------|
| `title` | `str` | Optional | 求人タイトル |
| `description` | `str` | Optional | 業務内容 |
| `required_skills` | `list[str]` | 必須（空配列可） | 必須スキル（タグ配列） |
| `preferred_skills` | `list[str]` | 必須（空配列可） | 歓迎スキル（タグ配列） |
| `ideal_profile` | `str` | Optional | 求める人物像 |
| `employment_type` | `EmploymentType` | Optional | enum |
| `location` | `str` | Optional | 勤務地 |
| `remote_work` | `RemoteWork` | Optional | enum |
| `rate_min` | `int` | Optional | 単価下限。単位: 万円/月 |
| `rate_max` | `int` | Optional | 単価上限。単位: 万円/月 |
| `min_experience_years` | `int` | Optional | 最低経験年数。単位: 年 |
| `position_level` | `PositionLevel` | Optional | enum |
| `industry_experience` | `str` | Optional | 業界経験 |
| `certifications` | `list[str]` | 必須（空配列可） | 資格要件 |

**CandidateParseResult（候補者）**

| フィールド | 型 | 必須/Optional | 単位・備考 |
|------------|----|--------------|-----------|
| `name` | `str` | Optional | 氏名 |
| `age` | `int` | Optional | 単位: 歳 |
| `nearest_station` | `str` | Optional | 最寄り駅 |
| `desired_rate` | `int` | Optional | 希望単価。単位: 万円/月 |
| `experience_years` | `int` | Optional | 経験年数。単位: 年 |
| `skills` | `list[str]` | 必須（空配列可） | スキル（タグ配列） |
| `certifications` | `list[str]` | 必須（空配列可） | 資格 |
| `work_history` | `str` | Optional | 職歴 |
| `education` | `str` | Optional | 学歴 |
| `self_pr` | `str` | Optional | 自己PR |

**単位の統一**

| 対象 | 単位 | 理由 |
|------|------|------|
| 単価（求人レンジ・候補者希望） | 万円/月（int） | 国内のSES・業務委託慣習。`80`=月80万円。丸め誤差なし |
| 経験年数 | 年（int） | 「3年半」等は四捨五入で int 化（プロンプトで明示） |

> 原文（`raw_text`）はスキーマに含めない。LLM に生成させると改変リスクがあるため、サーバが受け取った入力をそのまま保持する。

**ollama `format` への JSON Schema 受け渡し時の留意点**

- `BaseModel.model_json_schema()` を `format` に渡す。
- Optional は JSON Schema 上 `anyOf: [{type: ...}, {type: null}]` になる。各 Optional フィールドの description に「該当情報がなければ null」と明記し、モデルが null を返せるようにする。
- 配列フィールドは null ではなく**空配列 `[]`** をデフォルトとし、description でも統一する（null と空配列の揺れを防ぐ）。
- enum は JSON Schema 上 `enum: [...]` 制約として渡るため、候補外の値を出しにくくなる。候補値は日本語ラベルで統一し、フロント表示・DB値と一致させる。
- 各 `Field(description=...)` は LLM への抽出指示としても機能するため、「何を抽出するか・正規化ルール・欠損時の扱い」を1〜2文で記載する。**ただし `gemma4:e4b` は description の読み取りが弱く、これだけに頼ると主要フィールドが空で返る**ため、抽出指示の主役は system プロンプト側の明示列挙とする（[07_model-characteristics](07_model-characteristics.md) 参照）。

#### (b) プロンプト設計

`system` / `user` で役割を分担する。

| プロンプト | 役割 | 内容 |
|-----------|------|------|
| `system` | 抽出ルール・正規化ルール・出力契約の固定 | 役割定義（採用情報の構造化アシスタント）、共通の抽出方針（推測禁止・記載なしは null・配列は空配列・日本語出力・単位統一）、「JSON Schema に従って出力する」旨 |
| `user` | 対象テキストの投入 | 「以下の{求人票/応募書類}を構造化してください」＋デリミタで囲んだ生テキスト |

**共通の抽出ルール（system に記載）**

1. 原文に明示された情報のみを抽出する。推測・補完・創作をしない。
2. 該当情報がないフィールドは `null`（配列フィールドは `[]`）にする。
3. スキルは1スキル1要素に分割する（例: 「Python/Django」→ `["Python", "Django"]`）。表記は原文の語をそのまま使い、勝手な言い換えをしない。
4. 単価は万円/月の整数に正規化する（「時給」「年収」表記がある場合のみ概算換算してよいが、不明確なら null）。
5. 経験年数は年（整数）。「3年半」等は四捨五入。
6. enum フィールドは指定された候補値のいずれかに正規化する。
7. 出力は日本語。指定 JSON Schema 以外のキー・説明文・前後テキストを出力しない。

**求人用・候補者用の分け方**

- system は2系統（求人用 / 候補者用）に分ける。共通ルール（推測禁止・null・単位・JSON のみ）を共通部品として持ち、対象固有の説明（求人=必須/歓迎スキルの区別、候補者=職歴と自己PRの違い）を末尾に付ける。
- **system プロンプトに抽出フィールドを明示列挙する**（名前・意味・enum候補値・正規化ルール・欠損時の扱い）。当初は「フィールド定義は `format` の JSON Schema が担保するので全列挙不要」とする想定だったが、`gemma4:e4b` では JSON Schema の description が抽出指示として機能せず主要フィールドが空になるため、実測に基づきプロンプト明示へ方針転換した（[07_model-characteristics](07_model-characteristics.md) 参照）。enum 候補値は `StrEnum` から動的生成してプロンプトに埋め込み、`schemas.py` の値とズレないようにする。
- スキルの分割粒度・単価単位・enum 正規化など判断が割れやすい点は、具体例つきでルール化する（例: 「Go (Gin / Go-chi)」→ `["Go", "Gin", "Go-chi"]`、「〜の開発経験」は技術名だけ取り出す）。

#### (c) 堅牢化・エラー処理

| 観点 | 方針 |
|------|------|
| パース/検証失敗の検出 | `client.chat()` の戻り（`message.content`）を `model_validate_json()` で検証。JSON 構文エラー / バリデーションエラーの双方を捕捉 |
| リトライ回数 | 最大3回（初回 + リトライ2回）。`.env` の `PARSE_MAX_RETRIES` で可変 |
| フィードバック | 有り。失敗時はリトライ用の追加 user メッセージに「直前出力が不正だった旨＋エラー要約＋JSON Schema に厳密に従う指示」を付与して再実行する |
| リトライ上限超過 | `HTTP 502 Bad Gateway`（外部LLM由来の失敗）を返す |
| タイムアウト | ollama は `OLLAMA_TIMEOUT`（既定120秒）。超過時はリトライ対象とし、最終的に上限超過すれば 502。claude は SDK 既定のタイムアウト |
| LLM 接続不可 | 接続例外・認証不可は `HTTP 503 Service Unavailable`（ollama 未起動 / claude の `ANTHROPIC_API_KEY` 誤りなどの明示）。リトライせず即時打ち切り |
| 出力の安定性 | ollama は `options` に **`temperature=0` を固定**。claude（Opus 4.8）は temperature を送れないため指定せず、決定論性は「同一スキーマ + プロンプト」で担保する。抽出・マッチングは決定論的であってほしい（[07_model-characteristics](07_model-characteristics.md) 参照） |

**エラーレスポンス形（共通）**

```json
{
  "detail": {
    "code": "PARSE_FAILED",
    "message": "構造化に失敗しました。テキストを確認して再試行してください。",
    "attempts": 3
  }
}
```

> `code` は `PARSE_FAILED` / `LLM_TIMEOUT` / `LLM_UNAVAILABLE` のいずれか。`format` で JSON Schema を固定しているため構文崩れは起きにくいが、enum 逸脱・型不一致・null 揺れは発生し得るため検証＋リトライを必須とする。

#### (d) 処理フロー・責務分割

**責務分担**

| レイヤー | 責務 |
|----------|------|
| `routers/`（API層） | 生テキスト受領 → services 呼び出し → 結果に `raw_text` を付与して返却。例外を HTTP ステータスへ変換。LLM の詳細やリトライは持たない |
| `services/`（サービス層） | プロンプト構築・Ollama 呼び出し・検証・リトライ・正規化。`*ParseResult` を返す |
| `schemas.py` | `JobParseResult` / `CandidateParseResult` / enum 定義 |
| `config.py` | モデル名・URL・タイムアウト・リトライ回数 |

**services 層の関数構成（案）**

| 関数 | シグネチャ（擬似） | 役割 |
|------|-------------------|------|
| 共通LLMユーティリティ | `structured_chat(system, user, schema) -> T` | `client.chat(format=schema.model_json_schema())` 呼び出し＋検証＋リトライ＋フィードバックを一手に担う汎用関数。構造化・マッチング双方で再利用 |
| 求人構造化 | `structure_job(raw_text) -> JobParseResult` | 求人用 system/user を組み立て `structured_chat` を呼ぶ |
| 候補者構造化 | `structure_candidate(raw_text) -> CandidateParseResult` | 候補者用 system/user を組み立て `structured_chat` を呼ぶ |

**処理フロー（`POST /jobs/parse` の例）**

```
1. router: 生テキストを受領
2. service: structure_job(raw_text)
     ├─ プロンプト構築（求人用 system + user）
     └─ structured_chat()
          ├─ client.chat(model, messages, format=schema, timeout)
          ├─ Pydantic 検証
          ├─ 失敗 → エラー要約を添えて最大3回リトライ
          └─ 上限超過 → 例外（502/503 へ変換）
3. router: JobParseResult を返す（未保存）
     └─ raw_text を結果に同梱して返却（フロントが保存時に再送）
```

**parse の未保存・原文の扱い**

- `parse` は DB に書き込まない。構造化結果のみ返す（副作用なし・冪等）。
- `raw_text` は LLM に生成させず、router がリクエスト入力をそのまま結果に同梱して返す（改変防止）。
- フロントは parse 結果をユーザー編集後、`POST /jobs` / `POST /candidates` で `raw_text` 含めて再送し、そこで初めて保存される。

> `.env`（pydantic-settings）の想定キー: `LLM_PROVIDER`（`ollama` | `claude`、既定 `ollama`）/ `OLLAMA_BASE_URL` / `OLLAMA_MODEL` / `OLLAMA_TIMEOUT` / `CLAUDE_MODEL`（既定 `claude-opus-4-8`）/ `CLAUDE_MAX_TOKENS` / `PARSE_MAX_RETRIES`。`config.py` の設定クラスに集約する（05_setup 参照）。`ANTHROPIC_API_KEY` は Settings に持たず anthropic SDK が環境変数から直接解決する。

### マッチング

保存済みの候補者データ × 求人要件を LLM に渡し、多軸スコア・必須要件チェック・サマリーを JSON で返させる。候補者登録時（`POST /candidates`）に `BackgroundTasks` で起動し、結果を `scores` テーブルに保存する（04_api 参照）。同じ候補者×求人の再評価は上書き保存する（冪等）。

#### 基本方針

| 項目 | 方針 |
|------|------|
| 入力 | 構造化済みの候補者データ（`CandidateParseResult` 相当）＋ 求人要件（`JobParseResult` 相当）。原文ではなく構造化データを渡す |
| LLM の責務 | 各軸スコア（0〜100）・必須要件ごとの yes/no 判定・強み/懸念点/確認事項の生成までを担う |
| コード側の責務 | 総合スコアの重み付け集計・必須充足率の算出はコード側で行う（LLM には総合スコアを出させない） |
| 言語 | 強み・懸念点・確認事項は日本語の文章で返す |
| 再評価 | 同一候補者×求人は上書き保存（`candidate_id` 単位で冪等） |

#### (a) Pydantic スキーマ定義（`schemas.py`）

LLM 出力検証用に `MatchResult` を定義する。総合スコア・必須充足率は LLM 出力に含めず、コード側で算出して `scores` に保存する。

**enum 定義**

| enum | 候補値 | 補足 |
|------|--------|------|
| `RequirementStatus`（必須要件の充足） | 充足 / 未充足 | 各必須要件ごとの yes/no 判定 |

**RequirementCheck（必須要件1件）**

| フィールド | 型 | 必須/Optional | 備考 |
|------------|----|--------------|------|
| `requirement` | `str` | 必須 | 必須要件のラベル（求人の `required_skills` の各要素） |
| `status` | `RequirementStatus` | 必須 | 充足 / 未充足 |
| `evidence` | `str` | Optional | 充足/未充足と判断した根拠（候補者データ中の該当箇所） |

**MatchResult（LLM 出力）**

| フィールド | 型 | 必須/Optional | 単位・備考 |
|------------|----|--------------|-----------|
| `skill_score` | `int` | 必須 | スキルスコア。0〜100 |
| `experience_score` | `int` | 必須 | 経験年数スコア。0〜100 |
| `industry_score` | `int` | 必須 | 業界経験スコア。0〜100 |
| `position_score` | `int` | 必須 | ポジションレベルスコア。0〜100 |
| `requirement_checks` | `list[RequirementCheck]` | 必須（空配列可） | 求人の各必須スキルに対する充足判定 |
| `strengths` | `str` | 必須 | 強み（日本語文章） |
| `concerns` | `str` | 必須 | 懸念点（日本語文章） |
| `interview_points` | `str` | 必須 | 面接での確認事項（日本語文章） |

> スコアは `0 <= x <= 100` の制約（`Field(ge=0, le=100)`）を付ける。`requirement_checks` は求人の `required_skills` 各要素に1対1で対応させる（プロンプトで指示）。

**コード側で算出する派生値（`scores` に保存）**

| 値 | 算出方法 |
|----|---------|
| 総合スコア | 4軸スコアの重み付け平均（重みは `config.py`、下記参照） |
| 必須充足率 | `requirement_checks` のうち `充足` 件数 / 全件（例: 3/5）。`必須要件チェック(JSON)` 列には `requirement_checks` をそのまま保存 |

#### (b) スコアリングルール

**総合スコアの算出（コード側）**

- 各軸スコア（0〜100）を `config.py` の固定重みで加重平均して総合スコア（0〜100）を出す。
- 重みは `.env` で可変。既定はスキル重視とする。

| 軸 | 既定重み |
|----|---------|
| スキル | 0.40 |
| 経験年数 | 0.20 |
| 業界経験 | 0.20 |
| ポジションレベル | 0.20 |

- 重みは合計1.0に正規化して扱う（`.env` で和が1.0でなくても正規化する）。

**必須要件チェック（LLM）**

- チェック項目は求人の `required_skills` から生成する。各スキルを1項目として、候補者データに照らして `充足` / `未充足` を判定させる。
- 必須充足率はコード側で `充足` 件数を数えて算出する（LLM には数えさせない）。

#### (c) プロンプト設計

`system` / `user` で役割を分担する。

| プロンプト | 役割 | 内容 |
|-----------|------|------|
| `system` | 評価ルール・スコアリング基準・出力契約の固定 | 役割定義（採用マッチングアシスタント）、各軸の評価観点、スコアの段階感（例: 必須スキルを大半満たす=高得点）、必須要件は1件ずつ判定、サマリーは日本語・簡潔、JSON Schema に従う旨 |
| `user` | 求人要件＋候補者データの投入 | 構造化済みの求人要件と候補者データをそれぞれデリミタで囲んで渡す。「この候補者を求人に対して評価してください」 |

**評価ルール要点（system に記載）**

1. 各軸は 0〜100 で評価し、判断は渡された構造化データに基づく（原文の創作はしない）。
2. `requirement_checks` は求人の必須スキル**全件**を漏れなく列挙し、各々を `充足`/`未充足` で判定する。
3. 充足/未充足の判断根拠（`evidence`）は候補者データ中の該当箇所を簡潔に示す。
4. 強み・懸念点・確認事項はそれぞれ日本語の文章で、候補者固有の内容にする（汎用的な決まり文句を避ける）。
5. 総合スコア・必須充足率は出力しない（コード側で算出する）。
6. 指定 JSON Schema 以外のキー・前後テキストを出力しない。

#### (d) 堅牢化・エラー処理

構造化と同じ `structured_chat` 汎用関数（検証＋リトライ＋フィードバック）を再利用する。

| 観点 | 方針 |
|------|------|
| 検証 | `MatchResult.model_validate_json()`。スコア範囲（0〜100）・必須フィールドをバリデーションで担保 |
| リトライ | 最大3回（`PARSE_MAX_RETRIES` を共用）。失敗時はエラー要約を添えて再実行 |
| タイムアウト | `OLLAMA_TIMEOUT` を共用。超過はリトライ対象 |
| バックグラウンド失敗時 | スコア算出は `BackgroundTasks` 内のため HTTP では返さない。失敗を `scores` に記録するか未保存のままにし、フロントはポーリングで「未算出」を検知する（04_api 参照）。リトライ上限超過時はスコアを保存しない |

> `requirement_checks` の件数が求人の `required_skills` 件数と一致しない場合もバリデーション対象とし、不一致ならリトライする。

#### (e) 処理フロー・責務分割

**責務分担**

| レイヤー | 責務 |
|----------|------|
| `routers/`（API層） | `POST /candidates` で候補者を保存し、`BackgroundTasks` にスコア算出を登録してすぐ返す。スコア算出ロジックは持たない |
| `services/`（サービス層） | 求人・候補者データの取得 → プロンプト構築 → `structured_chat` で `MatchResult` 取得 → 総合スコア・必須充足率を算出 → `scores` に上書き保存 |
| `schemas.py` | `MatchResult` / `RequirementCheck` / enum 定義 |
| `config.py` | 各軸の重み・モデル名・タイムアウト・リトライ回数 |

**services 層の関数構成（案）**

| 関数 | シグネチャ（擬似） | 役割 |
|------|-------------------|------|
| マッチング実行 | `match_candidate(candidate_id) -> None` | 候補者と紐づく求人を取得し、評価・算出・保存までを担う。`BackgroundTasks` から呼ばれる |
| LLM 評価 | `evaluate_match(job, candidate) -> MatchResult` | マッチング用 system/user を組み立て `structured_chat` を呼ぶ |
| 総合スコア算出 | `compute_total_score(result, weights) -> int` | 4軸スコアを加重平均（純粋関数・テスト容易） |

**処理フロー（候補者登録時）**

```
1. router: POST /candidates で候補者を保存（構造化済みデータ＋raw_text）
2. router: BackgroundTasks に match_candidate(candidate_id) を登録 → 即レスポンス
3. service: match_candidate()
     ├─ candidate と job を取得
     ├─ evaluate_match(job, candidate)
     │     └─ structured_chat() で MatchResult（各軸スコア・必須判定・サマリー）
     ├─ compute_total_score()（重み付け平均）
     ├─ 必須充足率を算出（充足件数 / 全件）
     └─ scores に上書き保存（candidate_id 単位で冪等）
4. フロント: ポーリング/再取得でスコア完了を検知しランキングへ（04_api 参照）
```

> `config.py` の想定キー（構造化と共用＋追加）: `MATCH_WEIGHT_SKILL` / `MATCH_WEIGHT_EXPERIENCE` / `MATCH_WEIGHT_INDUSTRY` / `MATCH_WEIGHT_POSITION`（`.env` で可変、和は正規化）。
