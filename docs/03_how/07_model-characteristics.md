# 使用モデルの特性と対処方針 (how)

本プロジェクトが使う LLM（Ollama 上の `gemma4:e4b`）で実際に観測した挙動と、構造化精度を出すための対処方針をまとめる。`services/llm.py` のプロンプト設計・パラメータ設定はここでの知見に基づく。

> 対象モデル: **`gemma4:e4b`**（Ollama, 約9.6GB。有効パラメータは4B級の軽量モデル）。`.env` の `OLLAMA_MODEL` で切り替え可能だが、既定はこのモデル。

## 要約（結論）

| 観点 | このモデルの特性 | 対処 |
|------|------------------|------|
| JSON Schema の `description` | **読み取りが弱い**。`format` に JSON Schema を渡しても、各 `Field(description=...)` の抽出指示を取りこぼし、`title`/`description`/`required_skills` などが軒並み null・空配列で返ることがある | **system プロンプトに抽出フィールドを明示列挙する**（スキーマ任せにしない） |
| 生成のブレ | `temperature` 未指定だと**同一入力でも結果が大きく変動**する（あるとき抽出され、あるとき null） | 抽出タスクは **`temperature=0` を固定**して決定論的にする |
| enum 制約 | `format` の `enum: [...]` 制約は概ね効く。候補外は出にくい | enum 候補値は system プロンプトにも明記して二重化（スキーマと値を一致させる） |
| タグ分割・粒度 | 指示が曖昧だと「〜の開発経験」のような**文章をそのままタグに入れる**／括弧内を分割しない／学位要件を冗長展開する | プロンプトで分割例・タグ化ルールを具体例つきで明示する |

## 観測した事象（調査ログ）

実在の求人サンプル2パターンで `structure_job` を検証して得た知見。

### 1. スキーマ description 依存では主要フィールドが空になる

`_JOB_SYSTEM_PROMPT` が「フィールドの全列挙はせず、判断が割れやすい点だけ書く」方針だった当初、`title`/`description`/`required_skills`/`preferred_skills` が **null・空配列で返る**ことが多発した。

- `temperature` 未指定だと実行ごとに結果がバラつき、たまたま埋まる回もあった（=ブレで隠れていた）。
- `temperature=0` に固定すると、**決定論的にほぼ空の JSON** が返ることが判明 → ブレではなく「モデルがスキーマの description を抽出指示として読めていない」のが本質と特定。

→ **対処**: system プロンプトに抽出対象フィールドを名前・意味・enum候補つきで列挙したところ、主要フィールドがすべて埋まるようになった。

### 2. タグの粒度がコントロールされない

フィールドが埋まるようになった後も、配列フィールドの粒度に問題が残った。

| 症状 | 例 | 対処 |
|------|----|------|
| 経験文をそのままタグ化 | `required_skills` に「Webアプリケーション開発経験」「リレーショナルデータベース…の設計・チューニング経験」が入る | 「『〜経験』の文章はタグにせず、含まれる技術名だけ取り出す」と明示 |
| 括弧内が未分割 | 「Go (Gin / Go-chi)」が1要素のまま | 「『Go (Gin / Go-chi)』→ `["Go", "Gin", "Go-chi"]`」と分割例を明示 |
| 学位要件の冗長展開 | 「データサイエンス、または情報理工学系の修士・博士号」が4要素（修士/博士×2系統）に膨張 | 「複数候補が並んでも冗長展開せず簡潔にまとめる」＋ certifications に寄せる |
| preferred と certifications の重複 | 歓迎要件中の学位が両方に入る | 「資格・学位は preferred に入れず certifications に入れる」と明示 |

### 3. 表現で表せない歓迎要件はフレーズ許容が必要

歓迎要件は「Next.js への移行経験」「スクラムマスター経験」のように**技術名タグに還元できない経験要件**が多い。短いタグ縛りだけだとこれらが落ちるため、`preferred_skills` は「技術名はタグ分割、技術名で表せない経験要件は簡潔なフレーズ1件1要素」の二段構えにした。

## 対処方針（実装への落とし込み）

`services/llm.py` で以下を実施している。

1. **`structured_chat` に `temperature=0` を固定**（`options={"temperature": 0}`）。抽出・マッチングともブレを排除する。
2. **`_JOB_SYSTEM_PROMPT` に抽出フィールドを明示列挙**する。各フィールドの意味・enum候補値・正規化ルール・欠損時の扱いを書き下す。
3. **enum 候補値は `StrEnum` から動的生成**してプロンプトに埋め込む（`schemas.py` の定義とプロンプト記載の値がズレないようにする）。
4. **タグ化ルールを具体例つきで明示**する（分割例・「経験文はタグにしない」・学位の扱いなど）。

> この方針は「`format` の JSON Schema があるからプロンプトにフィールド全列挙は不要」という当初の想定（[02_ai.md](02_ai.md) の旧記述）を、実測に基づいて**覆した**もの。小型モデルでは JSON Schema 制約とプロンプト明示を**併用**する前提でいく。

## 他モデルへ切り替える場合の注意

- `.env` の `OLLAMA_MODEL` を上位モデル（例: `gemma4:12b`）にすれば description 依存の取りこぼしは軽減する可能性が高いが、ロードが重く検証に時間がかかる。本プロジェクトは軽量な `e4b` 前提で、**プロンプト側で精度を担保する**方針を採る。
- 上位モデル・別系統モデルに替えても `temperature=0` 固定とフィールド明示は害にならないため、そのまま維持してよい。
- 候補者構造化（`structure_candidate`）・マッチング（`evaluate_match`）も同じモデルを使うため、**同様の対処（フィールド明示 + `temperature=0`）を踏襲する**こと。実装時にこのドキュメントの方針を適用する。
